# Расширения каналов: направление P1 «песочница» (черновик, не реализовано)

P0 даёт **доверенные пути + проверка manifest + изоляция сбоев одного плагина**; код расширений разделяет адресное пространство с главным процессом. Ниже — возможные эволюции для оценки и планирования.

## Цели

Сохранить **семантику `ChannelPlugin`**, но вынести ненадёжный или рискованный inbound/outbound из главного процесса, сужая поверхность атаки.

## Вариант A: дочерний процесс‑адаптер

- В главном процессе остаётся тонкий **RPC‑клиент**; логика расширения в **дочернем** Node/Bun, обмен JSON по `stdio` или локальному сокету.
- Вызовы вроде `ChannelPlugin.outbound.sendText` на главном сериализуются в RPC; в дочернем вызывается реальный SDK.
- **Плюсы**: изоляция на уровне ОС, лимиты CPU/RAM (частично по платформам).  
- **Минусы**: задержка, сложность деплоя, синхронизация жизненного цикла с выходом из десктопа.

## Вариант B: потоки Worker

- Чистые вычисления или проверки без сети — в `worker_threads` (если уровень поддержки Bun достаточен).
- **Ограничение**: многие IM‑SDK требуют главный поток или native‑модули — часто всё равно нужен вариант A.

## Вариант C: V8 Isolate / класс `isolated-vm`

- Лёгкая изоляция в одном процессе; нужна оценка **совместимости с Bun** и доступности Node API.
- Подходит для **очень ограниченных** скриптов, не для полноценных официальных SDK.

## Набросок интерфейса (RPC)

```text
Главный процесс                    Дочерний процесс расширения
  |  spawn(channel-plugin.json)        |
  |----------------init--------------->|
  |<-------------ready-------------------|
  |  outbound.sendText(payload) ------> |
  |<------------- result --------------|
```

Конверт сообщений: `correlationId`, `channelId`, `method`, `payload`; ошибки с `code` + `message` (без утечки секретов).

## Критерии приёмки (на будущее)

- Падение дочернего процесса не валит главный; при выходе главного — SIGTERM с таймаутом и при необходимости SIGKILL.  
- Настраиваемые таймауты и квоты RPC (размер сообщения, QPS).

Текущий этап ориентируется на документацию и конфигурацию P0; внедрение схем с этой страницы — отдельный OpenSpec / дизайн‑ревью.
