# Расширения каналов (динамическая загрузка)

## Модель доверия P0 (обязательно к прочтению)

Расширения подключаются через динамический **`import()`** Bun в **том же процессе**, что и главный — **без** изоляции памяти как у полноценной песочницы. Устанавливайте и настраивайте **только доверенные** расширения (проверенный источник, аудит кода). Без настроенных `roots` расширения не загружаются. **Feishu** лежит в репозитории в `extensions/feishu/` и по умолчанию подхватывается из `config/channel-extensions.json` с `enabled: ["feishu"]` (в UI источник — «расширение»).

Приоритет и конфликты:

- **Встроенный** канал только WebUI; Feishu и прочие — через загрузчик расширений. Расширение **не может переопределить** уже зарегистрированный тот же `id` (встроенный `webui` регистрируется раньше) — при конфликте пакет пропускается, запись попадает в `errors` ответа `GET /api/channels`.
- Если два пакета объявляют один `id`, **побеждает первый успешно зарегистрированный**, второй пропускается.

## Структура пакета

Каждый плагин — подкаталог; родитель перечислен в `roots`:

```text
<root>/
  my-plugin/
    channel-plugin.json
    plugin.ts        # или .js, см. поле main
```

### channel-plugin.json

| Поле | Описание |
|------|----------|
| `id` | Уникальный id, должен совпадать с `ChannelPlugin.id` из фабрики |
| `name` | Отображаемое имя |
| `version` | Строка версии |
| `main` | ESM‑вход относительно каталога плагина; не абсолютный путь и без `..` |
| `capabilities` / `permissions` | Опционально, зарезервировано |

### Входной модуль

Нужен **default export** или именованный **`createChannelPlugin`** — фабрика, возвращающая `ChannelPlugin` или `Promise<ChannelPlugin>`.

Интерфейс: `src/channels/types.ts` (обязательны `config`, `outbound`, `status`; для долгоживущих соединений рекомендуется `setup`).

## Конфигурация

Сливаются два источника (оба есть — объединяются `roots`; **`enabled` с приоритетом у `~/.squid/channel-extensions.json`**):

1. `squid/config/channel-extensions.json` (можно создать по образцу `config/channel-extensions.example.json`)
2. `~/.squid/channel-extensions.json`

Поля:

- **`roots`**: `string[]`, каждый элемент — родительский путь с **несколькими** подкаталогами плагинов. Может быть абсолютным или относительным **корню репозитория squid**.
- **`enabled`** (опционально): если опущено или `null`, пробуются все прошедшие валидацию кандидаты; `[]` — не грузить расширения; непустой массив — **только** перечисленные `id`.

### Каталог пользователя `~/.squid/extensions` (не обязательно указывать в roots)

Если существует **`~/.squid/extensions`**, он **автоматически** добавляется как дополнительный корень сканирования (нет каталога — тихо игнорируется). Личные плагины, например `~/.squid/extensions/my-plugin/channel-plugin.json`. Загрузка по-прежнему ограничена **`enabled`** (при дефолте только `feishu` свой `id` нужно добавить в `~/.squid/channel-extensions.json` или проектный конфиг).

После изменения конфигурации нужен **перезапуск** хоста.

## Пример

В репозитории есть `extensions/example-echo-channel/`. В `config/channel-extensions.json`:

```json
{
  "roots": ["extensions"],
  "enabled": ["echo-demo"]
}
```

После перезапуска в боковой панели «Каналы» появится расширение `echo-demo` с источником «расширение».

## API

- `GET /api/channels` возвращает `{ "channels": [...], "errors": [...] }`. У канала есть `source`: `"builtin"` | `"extension"`. `errors` — некритичные ошибки сканирования/загрузки (без секретов).

## Локальная отладка

1. Создайте подкаталог в `roots` с `channel-plugin.json`.  
2. Для TypeScript убедитесь, что загрузка идёт через **Bun** (текущий бэкенд десктопа).  
3. Смотрите логи `[ChannelExtensions]` в консоли и оранжевый баннер в UI.

## Очередь «сессия занята» и ответ в тот же чат (без расширения `QueuedCommand`)

Как у Feishu / Telegram: если новому каналу нужно **после выполнения из очереди отправить ответ в тот же диалог**:

1. В **`setup.initialize`** расширения, если в контексте фабрики есть **`ctx.taskAPI`** (хост передаёт его в `initializeBuiltinChannels(taskAPI)`), вызовите **`registerXxxSquidBridge(ctx.taskAPI)`** (или эквивалент), внутри моста — **`taskAPI.addChannelQueuedCompleteHandler(...)`**, отправляйте сообщение только если `cmd.channelReply?.channelId === '<ваш channel id>'`; в **`setup.cleanup`** — отписка через возвращаемую функцию. **Хосту не нужно** импортировать `registerXxxSquidBridge` для каждого канала.
2. При занятости сессии вызывайте **`enqueueFromRequest(..., { channelReply: { channelId: '<как выше>', chatId: '<ключ маршрутизации>' } })`**. `chatId` — строка, семантику определяет канал.

Типы: **`ChannelQueueReply`** в `src/utils/messageQueueManager.ts`. Не добавляйте в ядро отдельные поля `xxxChatId`.

## Связь со встроенным вкладом

- **Встроенные** каналы по-прежнему можно добавлять PR в `src/channels` и регистрировать в `initializeBuiltinChannels`.  
- **Расширения** — для приватных или экспериментальных каналов без правки ядра; ответственность за безопасность на стороне развёртывания и источника кода.
