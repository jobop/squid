# Channel 拡張：P1 サンドボックス方向（スケッチ、未実装）

P0 は**信頼パス + manifest 検証 + 単一プラグイン失敗の分離**にとどまり、拡張コードはメインプロセスとアドレス空間を共有します。以下は将来のレビューと優先度付け用の任意案です。

## 目標

`ChannelPlugin` の**意味を変えず**、信頼できない、または高リスクな出站／入站処理をメインプロセス外へ移し、漏えい面を縮小する。

## 案 A：子プロセスアダプタ

- メインプロセスは薄い **RPC クライアント**のみ保持。拡張ロジックは Node/Bun の**子プロセス**で動き、`stdio` またはローカルソケットで JSON メッセージをやり取り。  
- `ChannelPlugin.outbound.sendText` などはメインプロセス側で RPC に直列化し、子プロセスで実 SDK を呼ぶ。  
- **利点**：OS レベルの分離、（一部プラットフォームで）CPU／メモリ制限。  
- **欠点**：遅延、配布の複雑さ、デスクトップ終了時の子プロセスライフサイクル同期。

## 案 B：Worker スレッド

- 純計算やネットワーク非依存の検証を `worker_threads` へ（Bun の対応状況が十分な場合）。  
- **制約**：多くの IM SDK はメインスレッドやネイティブモジュールに依存し、結局子プロセス案が必要になりがち。

## 案 C：V8 Isolate / `isolated-vm` 系

- 単一プロセス内の軽量分離。**Bun 互換性**と Node API 可用性を評価する必要あり。  
- **極めて制限された**スクリプト拡張向け。公式大型 SDK を直接載せる用途には不向き。

## インタフェーススケッチ（RPC）

```text
メインプロセス                    拡張子プロセス
  |  spawn(channel-plugin.json)    |
  |----------------init----------->|
  |<-------------ready--------------|
  |  outbound.sendText(payload) --> |
  |<------------- result ----------|
```

メッセージエンベロープに `correlationId`、`channelId`、`method`、`payload` を含める。エラーは `code` + `message`（マスク済み）。

## 受け入れの目安（将来）

- 子プロセスがクラッシュしてもメインプロセスが巻き添えにならない。メインプロセス終了時は子へ SIGTERM、期限後に SIGKILL。  
- 拡張ごとの RPC タイムアウトとクォータ（メッセージサイズ、QPS）を設定可能。

現行マイルストーンは P0 のドキュメントと設定が正です。本ページの案を実装する前に、別途 OpenSpec／設計レビューが必要です。
